LAST STEPS TO GET F1 STORIES TO A+

Objective

Bring the codebase, visuals, and performance from the current B to B+ range to an A+ production standard.

The target is not only better Lighthouse numbers. The target is a product that feels professionally operated: fast first paint, stable layout, polished mobile UI, clean deploy surface, compliant analytics behavior, automated regression gates, and no public clutter.


A+ Acceptance Bar

Use these as release gates. Do not call the work complete until all are true.

1. Production Lighthouse, mobile, median of 3 runs:
   - Home: Performance >= 95, Accessibility >= 98, Best Practices = 100, SEO = 100.
   - Blog index: Performance >= 95, Accessibility >= 98, Best Practices = 100, SEO = 100.

   - LCP <= 2.5s on home, blog index, and standings.
   - CLS <= 0.02 on home, blog index, and standings.
   - INP <= 200ms.
   - TBT <= 100ms in Lighthouse.

   - No clipped tags, buttons, category pills, cards, chart labels, or nav items on 390px, 768px, 1440px, and 1920px widths.

   - No private task files served from the public site.

   - No source-only files such as `.docx`, draft `.txt`, original large `.png`, or original `.jpg` files in the deploy artifact unless they are intentionally linked public assets.

5. Automation:
   - One command verifies build, tests, budget, and Lighthouse budgets locally.
   - CI runs the same checks on pull requests or pushes to main.
   - Performance budget covers route-level Lighthouse results, not only file byte sizes.

PHASE 1: Fix Standings Performance And CLS

Goal: make standings feel like a professional data dashboard. This is the biggest production defect because CLS 1.036 is not acceptable.


1. Fix minified ES module imports

Problem:

`standings/standings.min.js` is minified, but it imports unminified modules:

`import ... from "./core/format.js"`

That makes the browser fetch unminified core files after the page already preloads the minified files. It wastes requests, makes preloads less useful, and creates inconsistent deploy behavior.

Files:

1. `scripts/build/minify.mjs`
2. `standings/standings.min.js`
3. `standings/core/*.min.js`
4. `standings/tabs/*.min.js`
5. `standings/index.html`

Implementation steps:

1. Add a helper in `scripts/build/minify.mjs` after JS minification:
   - It should rewrite static ESM import specifiers ending in `.js` to `.min.js`.
   - It should rewrite static export specifiers ending in `.js` to `.min.js`.
   - It should rewrite dynamic imports where the string literal is directly visible.
   - It must only rewrite relative imports beginning with `./` or `../`.
   - It must not rewrite external URLs.

2. Apply the rewrite only to browser-shipped ESM modules:
   - files under `standings/`
   - any future browser module that imports sibling modules

3. Keep source files readable.
   - Do not change source imports in `standings/standings.js` to `.min.js`.
   - Source files should continue importing `.js`.
   - The build step should produce deploy-ready minified imports.

4. Run:
   `npm run build:assets`

5. Verify the generated file:
   `rg -n "from\\s+[\"']\\./core/.*\\.js|import\\([\"']\\./tabs/.*\\.js" standings/standings.min.js`

   Expected:
   - No raw `.js` imports from `standings.min.js`.
   - Imports should point to `.min.js` files.

6. Verify browser network:
   - Open `/standings/` in a clean browser context.
   - Confirm no requests for:
     - `/standings/core/format.js`
     - `/standings/core/teams.js`
     - `/standings/core/drivers-meta.js`
     - `/standings/core/cache.js`
     - `/standings/core/fetchers.js`

Acceptance:

1. Standings page requests only minified browser JS files.
2. Lighthouse no longer reports unminified JavaScript on standings.
3. No duplicate preload/import mismatch remains.

   - `DM Sans Fallback`

   - Do not apply blindly to every font.

6. Lock stable dimensions for standings above-the-fold containers.

   - Tabs row should reserve final height and horizontal overflow.
